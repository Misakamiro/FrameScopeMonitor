The host uses:

- `Microsoft.Web.WebView2.WinForms.WebView2`
- `CoreWebView2Environment.CreateAsync`
- `CoreWebView2.PostWebMessageAsJson`
- `CoreWebView2.WebMessageReceived`

The current hosted page is an embedded minimal HTML bridge surface. This is intentional for this backend-bridge round: no React/Vite visual migration is included. The page is only a compatibility surface for real bridge messages and smoke validation.

Navigation is restricted. The host allows only the embedded `data:` navigation generated by `NavigateToString` and `about:` internal navigation. Arbitrary external navigation is blocked.

## Bridge Contract

Request:

```json
{ "requestId": "abc", "type": "config.get", "payload": {} }
```

Response:

```json
{ "requestId": "abc", "type": "response", "ok": true, "payload": {}, "error": null }
```

Error response:

```json
{ "requestId": "abc", "type": "response", "ok": false, "payload": {}, "error": { "code": "unsupported_request", "message": "..." } }
```

Event:

```json
{ "type": "event.processesRefreshed", "payload": {}, "sentAt": "2026-05-20T13:11:56.9693352+08:00" }
```

Contract invariants implemented:

- Every parsed request receives exactly one response.
- Missing `requestId` returns `missing_request_id`.
- Unknown request types return `unsupported_request`.
- Handler failures return `handler_failed` and also publish `event.error`.
- `processes.refresh` returns `accepted` immediately and pushes `event.processesRefreshed` later.

## Validation And Safety

C# owns all system operations:

- Config reads/writes use `FrameScopeConfigStore`.
- Config write path is fixed by host options and validated under the app root.
- Process refresh uses safe per-process exception handling from `FrameScopeProcessPicker`.
- The frontend never executes a process, opens a path, or chooses a config file path.

## MSB3277 / WindowsBase Handling

The main application is still compiled by `csc.exe` in `build.ps1`, not by an SDK-style project. Therefore the main build does not run MSBuild assembly conflict resolution and did not emit an `MSB3277 WindowsBase` warning.

If the standalone SDK spike emits an `MSB3277` warning around `WindowsBase`, it is treated as a spike-only transitive WPF warning. The shipped path avoids it by referencing only WebView2 Core + WinForms and excluding WebView2 WPF.
