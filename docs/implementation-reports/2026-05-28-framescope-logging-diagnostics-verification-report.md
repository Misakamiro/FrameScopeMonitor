## 本轮修正

1. 新增 `src\core\FrameScopeLoggingPolicy.cs`，集中定义：
   - `EnableVerboseLogs` 是否允许详细日志。
   - `EnablePerformanceDiagnosticsLogs` 是否允许性能诊断日志。
   - `AutoGenerateDiagnosticReport` 是否应在当前 monitor 结果下真正生成自动诊断文件。
2. watcher 不再把自动诊断等同于“每次完成都生成”：
   - 健康 full report、有帧数据、`FrameCaptureStatus=captured` 时跳过自动诊断。
   - 非 0 exit、`Phase=error`、`ReportError`、`ReportKind=diagnostic`、非 `captured` 的帧捕获状态会触发自动诊断。
3. watcher 到 monitor-session 的参数传递现在使用真实策略：
   - `--EnableVerboseLogs true/false`
   - `--EnablePerformanceDiagnosticsLogs true/false`
4. monitor-session 增加受控日志：
   - verbose：session 创建、目标选择、子进程启动等运行过程。
   - perf：session 创建耗时、目标等待耗时、子进程启动耗时、capture loop 耗时、session 总耗时。
5. report generation 增加受控性能诊断：
   - `report-generation-ms`、exit、frames、process/system sample count、report kind。
6. 新增 `tests\FrameScopeLoggingPolicyTests.cs`，并把它加入 `tests\Build-FrameScopeTests.ps1`。
7. `build.ps1` 已把 `src\core\FrameScopeLoggingPolicy.cs` 编译进主程序。

## 关键行为结论

1. 默认日志到底记录哪些内容？
   - 默认日志不是“只写 error/warn”这么窄；它会写关键生命周期事件，例如 `native-watcher-start`、`monitor-start`、`report-generate-start`、`report-generate-complete`、`monitor-complete`，以及失败/异常/跳过类事件。
   - 默认不会持续写普通 verbose、perf、debug 噪声。场景验证中默认 watcher 日志只有 1 行启动事件，`verbose/perf/debug` 噪声检查为 false。
2. 详细日志是否真实有效？
   - 是。关闭时不出现 `verbose` 运行过程；开启时出现 `verbose monitor-session-created`、`verbose monitor-target-selected`、`verbose monitor-children-started`。
3. 性能诊断日志是否真实有效？
   - 是。关闭时不出现 `perf`；开启时出现 monitor-session、target wait、child start、capture loop、session total，以及 report generation 耗时类信息。
4. 自动生成诊断文件是否真实有效？
   - 是。健康 full report 正常路径不会自动生成诊断目录；失败/异常路径会写 `diagnostic-report-queued` 并生成诊断文件。
   - WebView2 smoke 中手动调用 `diagnostics.generate` 会生成诊断报告，这是手动诊断动作，不属于自动诊断误触发。
5. 是否做了修正？
   - 是。修正集中在日志策略、watcher 自动诊断 gating、monitor-session/report generation 的 verbose/perf gating，以及新增回归测试。
6. 是否建议进入复测窗口？
   - 建议进入复测窗口。当前源码级、合成目标、前端、C#、WebView2 smoke 和残留进程检查均已通过。下一窗口建议继续保持“不启动 BF6/真实游戏”的前置约束，先做安装包/已安装版验证时再单独授权。
