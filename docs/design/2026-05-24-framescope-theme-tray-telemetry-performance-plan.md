## Current Code Observations

### React Theme / Token / Config / Save Structure

Current frontend structure:

- `src\frontend\src\main.tsx` imports `tokens.css` and `global.css`, then renders `App` inside `MotionConfig`.
- `src\frontend\src\theme\tokens.css` defines primitive, semantic, and component CSS custom properties under `:root`.
- `src\frontend\src\styles\global.css` applies `--fs-surface-window` and `--fs-text-primary` at `html`, `body`, and `.app-root`.
- `src\frontend\src\App.tsx` owns page routing and passes the shared `bridgeState` into pages.
- `src\frontend\src\pages\SettingsPage.tsx` reads `bridgeState.config.data?.config`, clones it into a `draft`, updates `draft` fields, and saves by calling `bridgeState.saveConfig(draft)`.
- `src\frontend\src\bridge\contract.ts` currently defines `FrameScopeConfig` with:
  - `PollIntervalMs`
  - `DataRoot`
  - `OpenReportOnComplete`
  - `EnableVerboseLogs`
  - `EnablePerformanceDiagnosticsLogs`
  - `AutoGenerateDiagnosticReport`
  - `LogRetentionDays`
  - `MaxLogDiskMb`
  - `MonitorScript`
  - `Targets`

Design implication:

- Theme must be persisted in `FrameScopeConfig`, not only in browser localStorage, because WebView2 is hosted inside a desktop app and the host already owns validated config persistence.
- The CSS token model should remain token-driven. Dark mode should override semantic/component tokens by theme attribute, not fork every component stylesheet.
- Settings must preserve unknown/new backend fields through full config round-trip. Any new field added to C# must also be added to TypeScript `FrameScopeConfig`, mock data, and contract tests.

### WebView2 Host / 默认入口 / WinForms Fallback Boundary

Current desktop entry behavior:

- `src\app\FrameScopeNativeMonitor.cs` handles `--monitor-session`, `--generate-diagnostic-report`, and `--watcher` before UI setup.
- Normal startup enables WinForms visual styles, checks WebView2 runtime, and calls `RunWebUi(args)`.
- Missing WebView2 runtime currently shows the runtime warning and exits with code `3`.
- `RunWebUi` creates `FrameScopeWebHostForm` with root/config/state/history paths, bridge host adapter, frontend path, smoke flags, evidence path, screenshot path, and timeout.
- `FrameScopeWebHostForm` maps `frontend` or `src\frontend\dist` to `https://app.framescope.local/index.html`; otherwise it loads embedded minimal bridge HTML.
- Existing older WinForms UI modules are not the default visual entry anymore, but tests and source boundaries still protect WinForms-era logic such as config, live refresh, and process cleanup.

Design implication:

- Do not replace backend modes or move `--watcher` and `--monitor-session`.
- Tray behavior belongs to `FrameScopeWebHostForm` / host window lifecycle, not to React alone.
- "WinForms fallback" for this plan means preserving the existing host, runtime self-test, embedded bridge fallback, and any legacy WinForms test seams. It does not mean changing default launch back to old UI.

### Bridge / Request / Event Structure

Current bridge request types:

- `state.snapshot`
- `config.get`
- `config.save`
- `processes.refresh`
- `monitor.start`
- `monitor.stop`
- `reports.list`
- `reports.open`
- `reports.openDirectory`
- `reports.regenerate`
- `diagnostics.generate`
- `targets.get`
- `targets.save`

Current event types:

- `event.status`
- `event.error`
- `event.processesRefreshed`
- `event.reportsChanged`
- `event.reportProgress`

Current bridge safety rules:

- `config.save` rejects `path` and `configPath`.
- report actions reject frontend-provided filesystem paths and use backend-issued `reportId`.
- monitor start/stop are long actions and finish through `event.status`.

Design implication:

- Theme and close behavior do not need a new `config.save` path. They should flow through existing `config.get` / `config.save`.
- Host window state may need events, because tray visibility and explicit quit are host-level behavior. Add small host-oriented request/event types only for user actions that cannot be represented as config save.

### Watcher / Monitor Lifecycle / watcher 生命周期

Current lifecycle:

- `monitor.start` calls host adapter `StartMonitor`.
- `StartMonitor` loads and normalizes config, saves it, checks `IsWatcherRunning`, then starts the same executable with `--watcher --config <ConfigPath>`.
- `RunNativeWatcher` scans enabled targets, starts `--monitor-session` for detected targets, tracks active monitors, writes watcher state, generates reports after completed runs, optionally opens reports, and logs `watcher-poll-ms` when performance diagnostics logging is enabled.
- `monitor.stop` calls host adapter `StopMonitor`, which uses `StopFrameScopeBackgroundProcesses`.
- `StopFrameScopeBackgroundProcesses` may kill watcher, monitor-session, sampler, and bundled PresentMon background processes.
- `--monitor-session` starts PresentMon, `FrameScopeProcessSampler.exe`, and `FrameScopeSystemSampler.exe`, waits until target exit or timeout, then writes summary/status and lets watcher/report orchestration continue.

Design implication:

- Closing the WebView2 window must not accidentally stop an active watcher unless the user explicitly chooses "exit software".
- Tray minimize must keep the host process alive, keep bridge state available after restore, and keep watcher/session/report generation untouched.
- Explicit quit must make a product decision: either stop FrameScope-owned background processes first or warn that monitoring is active. The design below chooses a safe confirmation path.

### Current System Telemetry Shape

Current `FrameScopeSystemSampler` writes `system-samples.csv` with:

```text
Time,SampleIndex,Cs2Running,TargetRunning,TotalCpuPct,CpuFrequencyMHz,CpuPerformancePct,AvailableMB,DiskAvgSecPerTransfer,DiskBytesPerSec,NetBytesPerSec,GpuUtilPct,GpuMemUtilPct,GpuTempC,GpuPState,GpuClockMHz,MemClockMHz,PowerW,VramUsedMiB,VramTotalMiB,ProcessCount
```

Current CPU frequency behavior:

- `FrameScopeSystemSampler.PerfCounters.cs` uses `Processor Information / Processor Frequency / _Total`.
- `FrameScopeReportGenerator.SystemData.cs` reads `CpuFrequencyMHz` and `CpuPerformancePct`, then calculates effective CPU frequency for report data.

Design implication:

- Per-core frequency should extend the system sampler or a sidecar owned by the system sampler.
- Voltage should not be presented as guaranteed. Windows built-in performance counters typically do not expose reliable per-core voltage. Voltage must be best-effort through a supported sensor provider, with unavailable state preserved rather than fake values.
- Do not add these values to chart UI in this feature. Store data and metadata only.

## 配置字段设计 / Configuration Field Design

### Additive Fields

Add these fields to `FrameScopeConfig`:

```json
{
  "ConfigSchemaVersion": 2,
  "ThemeMode": "system",
  "CloseWindowBehavior": "minimize-to-tray",
  "TrayEnabled": true,
  "CpuTelemetry": {
    "CollectPerCoreFrequency": true,
    "CollectCpuVoltage": true,
    "PerCoreSampleIntervalMs": 1000,
    "VoltageProvider": "auto"
  },
  "PerformanceBaseline": {
    "Enabled": false,
    "LastBaselineId": ""
  }
}
```

Recommended C# model:

```csharp
public int ConfigSchemaVersion { get; set; }
public string ThemeMode { get; set; }
public string CloseWindowBehavior { get; set; }
public bool TrayEnabled { get; set; }
public FrameScopeCpuTelemetryConfig CpuTelemetry { get; set; }
public FrameScopePerformanceBaselineConfig PerformanceBaseline { get; set; }
```

Recommended TypeScript model:

```ts
export type FrameScopeThemeMode = "light" | "dark" | "system";
export type FrameScopeCloseWindowBehavior = "exit" | "minimize-to-tray";

export interface FrameScopeCpuTelemetryConfig {
  CollectPerCoreFrequency: boolean;
  CollectCpuVoltage: boolean;
  PerCoreSampleIntervalMs: number;
  VoltageProvider: "auto" | "disabled" | "wmi" | "sensor";
}

export interface FrameScopePerformanceBaselineConfig {
  Enabled: boolean;
  LastBaselineId: string;
}
```

### Defaults

Default values:

- `ConfigSchemaVersion`: `2`
- `ThemeMode`: `"system"`
- `CloseWindowBehavior`: `"minimize-to-tray"`
- `TrayEnabled`: `true`
- `CpuTelemetry.CollectPerCoreFrequency`: `true`
- `CpuTelemetry.CollectCpuVoltage`: `true`
- `CpuTelemetry.PerCoreSampleIntervalMs`: `1000`
- `CpuTelemetry.VoltageProvider`: `"auto"`
- `PerformanceBaseline.Enabled`: `false`
- `PerformanceBaseline.LastBaselineId`: `""`

Rationale:

- `system` theme avoids surprising existing users.
- `minimize-to-tray` matches the requested new behavior but still lets users choose direct exit.
- Frequency collection is low-risk if done at slow interval.
- Voltage is enabled as a request, but provider must report unavailable instead of blocking capture.
- Performance baseline is not always-on because it is diagnostic overhead.

### Compatibility With Old Configs

This is the `config schema` extension point for this feature set.

`FrameScopeConfigStore.Normalize` should:

1. Treat missing fields as defaults.
2. Clamp invalid string enums:
   - invalid `ThemeMode` -> `"system"`
   - invalid `CloseWindowBehavior` -> `"minimize-to-tray"`
   - invalid `VoltageProvider` -> `"auto"`
3. Ensure `CpuTelemetry` and `PerformanceBaseline` objects are never null after normalization.
4. Clamp `PerCoreSampleIntervalMs`:
   - minimum `500`
   - default `1000`
   - recommended maximum `5000`
5. Keep existing fields and targets unchanged.
6. Preserve hidden intervals when `BuildConfigFromEditableTargets` merges target edits.
7. Save normalized config through the existing `FrameScopeConfigStore.Save`.

No migration file is needed. The first successful `config.save` or monitor start can write the normalized schema.

### Settings Save Rules

- React `SettingsPage` continues to edit `FrameScopeConfig` as a draft.
- `ThemeMode`, `CloseWindowBehavior`, `TrayEnabled`, and `CpuTelemetry` fields must be shown in Settings and included in the saved `draft`.
- `config.save` continues to reject `path` and `configPath`.
- `targets.save` must preserve new global fields when it rebuilds config from target rows.
- Mock preview data must include these fields so frontend tests do not drift.

## bridge/request/event 变化 / Bridge Request Event Changes

### Config Payload

No separate theme endpoint is required. `config.get` and `config.save` should include the new fields through `FrameScopeConfig`.

`state.snapshot` should add host state fields:

```json
{
  "host": {
    "windowVisible": true,
    "trayAvailable": true,
    "closeWindowBehavior": "minimize-to-tray",
    "themeMode": "system",
    "resolvedTheme": "dark"
  }
}
```

Rationale:

- `resolvedTheme` lets the React app apply the real host/system decision immediately.
- `windowVisible` and `trayAvailable` are host facts, not config facts.

### New Request Types

Add only host-owned actions:

```ts
export type BridgeRequestType =
  | existing
  | "host.window.show"
  | "host.window.hideToTray"
  | "host.app.exit";
```

Request semantics:

| Request | Caller | Behavior |
|---|---|---|
| `host.window.show` | Tray menu or future UI command | Restore form, show in taskbar, focus window, keep watcher unchanged. |
| `host.window.hideToTray` | Window close handler or future UI command | Hide form, set tray icon visible, keep watcher unchanged. |
| `host.app.exit` | Tray menu Exit, window X when behavior is exit | Run explicit exit path. If active monitoring exists, require confirmation before stopping background processes or leave watcher running based on final product decision. |

For this feature, tray menu can call C# methods directly because it is host UI. The bridge requests are still useful for smoke tests and future UI buttons.

### New Event Types

Add:

```ts
export type BridgeEventType =
  | existing
  | "event.hostWindowChanged"
  | "event.themeChanged";
```

Payloads:

```json
{
  "visible": false,
  "inTray": true,
  "closeWindowBehavior": "minimize-to-tray",
  "reason": "window-close"
}
```

```json
{
  "themeMode": "system",
  "resolvedTheme": "dark",
  "source": "config-save|system-change|startup"
}
```

System theme changes:

- C# host should listen for Windows theme change only if practical without large complexity.
- React should also handle `matchMedia("(prefers-color-scheme: dark)")` for immediate visual update when `ThemeMode = "system"`.
- The source of truth remains saved config plus resolved runtime theme.

## UI 交互 / UI Interaction Design

### Theme Application

React root should apply:

```html
<html data-theme="light">
```

or

```html
<html data-theme="dark">
```

Implementation approach:

1. Add a small theme controller hook, for example `src\frontend\src\theme\useFrameScopeTheme.ts`.
2. It reads `bridgeState.config.data?.config.ThemeMode`.
3. If `"system"`, resolve with `window.matchMedia("(prefers-color-scheme: dark)")`.
4. Set `document.documentElement.dataset.theme`.
5. Keep reduced-motion logic separate.

CSS approach:

- Keep existing light tokens in `:root`.
- Add `:root[data-theme="dark"]` overrides for semantic and component tokens.
- Avoid duplicating page/component CSS.
- Keep focus, warning, success, error, diagnostics colors accessible in both themes.

Dark token families should not become a one-note dark blue palette. Use neutral dark surfaces with distinct sky/mint/amber/rose/violet/teal state colors.

### Settings Page

Add a new Settings group: `Appearance and window behavior`.

Controls:

- Theme segmented control:
  - Light
  - Dark
  - Follow system
- Close window behavior segmented control:
  - Exit software
  - Minimize to tray
- Tray enabled toggle:
  - default on
  - disabled only if a future environment cannot show tray icon

Add a new Settings group or subsection: `Telemetry capture`.

Controls:

- CPU per-core frequency toggle.
- CPU voltage toggle.
- Per-core sample interval numeric input, default `1000 ms`.
- Voltage provider display or select:
  - `Auto`
  - `Disabled`
  - provider-specific options only after a provider is implemented.

Visible copy should stay product-facing:

- Do not show internal bridge names.
- Do not expose "schema version" to users.
- Do not imply voltage is guaranteed. Use a short availability line after a capture has data, not during settings.

### Window X Behavior

When user clicks X:

1. Load current normalized config from host memory or disk.
2. If `CloseWindowBehavior = "minimize-to-tray"` and `TrayEnabled = true`:
   - cancel form closing
   - hide window
   - show tray icon
   - keep watcher/monitor-session/report generation running
   - emit/log host window hidden event
3. If `CloseWindowBehavior = "exit"`:
   - if watcher/monitor-session/samplers are not running, close normally
   - if monitoring is active, show confirmation:
     - Exit and stop monitoring
     - Cancel
   - do not silently kill capture from an X click

Reason:

- A user may close the window while a game capture is active. Silent shutdown would corrupt or truncate the run.
- Minimize-to-tray should be reversible and should not change monitoring state.

### Tray Menu

Right-click menu:

- Show window
- Exit

Behavior:

- Show window:
  - `Show()`
  - `WindowState = Normal` if minimized
  - `ShowInTaskbar = true`
  - `Activate()`
  - keep tray icon visible or hide it based on final UX choice. Recommended: keep tray icon visible while app is running so users always have a stable control point.
- Exit:
  - same explicit exit path as window X with exit behavior.
  - if monitoring active, confirm before stopping.
  - after confirmed exit, dispose tray icon and WebView2, then close.

Tray icon lifetime:

- Create once in `FrameScopeWebHostForm`.
- Dispose in form dispose/close path.
- Never create duplicate icons after hide/show cycles.

## 托盘状态机 / Tray State Machine

### States

| State | Window | Tray Icon | Watcher / Monitor | Meaning |
|---|---|---|---|---|
| `VisibleIdle` | visible | visible | no watcher | App is open, not monitoring. |
| `VisibleWatcherIdle` | visible | visible | watcher running, no active monitor | App is open, scanning for targets. |
| `VisibleMonitoring` | visible | visible | watcher plus monitor-session/samplers | App is open during capture. |
| `TrayIdle` | hidden | visible | no watcher | User closed/minimized to tray while idle. |
| `TrayWatcherIdle` | hidden | visible | watcher scanning | User hid window while watcher scans. |
| `TrayMonitoring` | hidden | visible | active capture | User hid window while game is being captured. |
| `ExitingClean` | closing | disposing | no active monitoring or confirmed stop | App exits normally. |
| `ExitBlocked` | visible or hidden | visible | active monitoring | Exit was requested but user cancelled or confirmation failed. |
| `HostFailed` | closing or visible error | disposed if closing | unchanged unless explicit stop | WebView2 failed or host exception occurred. |

### Transitions

| Trigger | From | To | Rules |
|---|---|---|---|
| Startup | none | `VisibleIdle` / `VisibleWatcherIdle` | Use `state.snapshot` and `IsWatcherRunning`. |
| `monitor.start` accepted/completed | visible/tray idle | watcher state | Start watcher only, do not change window visibility. |
| target detected | watcher idle | monitoring | Watcher starts `--monitor-session`; UI follows snapshot/event refresh. |
| monitor completes | monitoring | watcher idle | Report generation and auto-open continue; tray state stays hidden if hidden. |
| `monitor.stop` completed | watcher/monitoring | idle | Explicit stop from UI or confirmed exit. |
| Window X with minimize-to-tray | visible states | matching tray state | Cancel close; hide form; no watcher changes. |
| Window X with exit and no monitoring | visible states | `ExitingClean` | Close form and dispose tray/WebView2. |
| Window X with exit and active monitoring, user cancels | visible states | `ExitBlocked` then previous state | No process changes. |
| Window X with exit and active monitoring, user confirms | visible monitoring | `ExitingClean` | Stop FrameScope background processes, then close. |
| Tray Show window | tray states | matching visible state | Restore form only. |
| Tray Exit with active monitoring, user cancels | tray states | previous tray state | No process changes. |
| Tray Exit with active monitoring, user confirms | tray monitoring | `ExitingClean` | Stop FrameScope background processes, then close. |
| WebView2 process failed | any visible/tray state | `HostFailed` or previous state with error | Do not stop watcher automatically; log and show recoverable message if possible. |

### Explicit Exit Policy

Recommended policy:

- Direct user exit means "exit FrameScope Monitor", including watcher and samplers, but only after confirmation if active capture exists.
- If only watcher scanning is running and no active monitor-session exists, exit can stop watcher without confirmation.
- If monitor-session/samplers/PresentMon are active, confirmation is required.

Implementation must distinguish:

- watcher PID exists
- active monitors in `framescope-watcher-state.json`
- sampler/PresentMon processes from `EnumerateFrameScopeBackgroundPids`

## CPU 采集方案 / CPU Per-Core Frequency And Voltage Collection Plan

### Data Flow Boundary

Do not put per-core telemetry directly into the chart data in this feature.

Recommended data flow:

```text
FrameScopeSystemSampler.exe
  -> system-samples.csv keeps existing aggregate columns
  -> cpu-core-samples.csv stores per-core frequency and voltage
  -> status.json / summary.json notes telemetry availability
  -> FrameScopeReportGenerator reads metadata only for manifest
  -> HTML report does not chart the new values yet
```

### New Artifact

Create:

```text
cpu-core-samples.csv
```

Recommended columns:

```text
Time,SampleIndex,ElapsedMs,CoreId,ProcessorInstance,FrequencyMHz,EffectiveFrequencyMHz,PerformancePct,VoltageV,VoltageSource,Available
```

Rules:

- One row per logical processor instance per slow telemetry sample.
- Use the same session time base as `system-samples.csv`.
- Sample interval defaults to `CpuTelemetry.PerCoreSampleIntervalMs`.
- Do not sample faster than `500 ms`.
- If voltage unavailable, write empty `VoltageV`, set `VoltageSource=unavailable`, and `Available=false` for voltage fields.
- Do not block system sampler if voltage provider fails.

### Frequency Provider

Primary:

- Windows Performance Counter category `Processor Information`.
- Enumerate instances excluding `_Total`.
- Read:
  - `Processor Frequency`
  - `% Processor Performance`

Derived:

- `EffectiveFrequencyMHz = FrequencyMHz * PerformancePct / 100` when both values are available.
- If `% Processor Performance` is unavailable, use raw `FrequencyMHz`.

Compatibility:

- If per-core counters fail, keep aggregate `system-samples.csv` unchanged and record metadata:
  - `CpuCoreTelemetryAvailable=false`
  - `CpuCoreTelemetryError=<redacted short reason>`

### Voltage Provider

Voltage should be treated as best-effort.

Provider order:

1. `disabled`: skip when config says disabled.
2. `auto`: try safe built-in provider first.
3. `wmi`: try WMI classes if available.
4. `sensor`: future provider hook if a bundled sensor library or approved dependency exists.

Important constraints:

- Do not add an always-on heavyweight third-party hardware sensor dependency without a separate decision.
- Do not require admin just for voltage capture.
- Do not fake voltage from VID, nominal voltage, or static hardware metadata.
- Do not fail monitor-session because voltage is missing.

Recommended first implementation:

- Implement per-core frequency first.
- Add voltage columns and availability metadata.
- Set voltage unavailable unless a reliable provider is proven during a baseline run.

### Report / Manifest Handling

In this feature:

- `FrameScopeReportGenerator` may count rows and add manifest fields:
  - `cpuCoreSampleCount`
  - `cpuCoreTelemetryAvailable`
  - `cpuVoltageAvailable`
  - `cpuVoltageSource`
- Do not add chart tabs, series, or visible report controls yet.
- Old reports without `cpu-core-samples.csv` must still load with counts as `0` and availability as `false`.

## 性能基线方案 / Performance Baseline Plan

Performance optimization must be evidence-led. No "looks faster" changes should be accepted.

### Baseline Artifacts

Create a baseline report directory per run:

```text
artifacts\performance-baselines\YYYYMMDD-HHMMSS\
```

Files:

- `baseline-summary.json`
- `frontend-webview2-smoke.json`
- `frontend-render-metrics.json`
- `watcher-poll.csv`
- `monitor-session-overhead.json`
- `sampler-overhead.json`
- `report-render-metrics.json`
- screenshots for desktop and reduced-motion if UI changed

### Metrics To Capture Before Optimization

Frontend / WebView2:

- Time from host form shown to `webview-ready`.
- Time from `webview-ready` to first `state.snapshot` success.
- Page switch timings for overview -> targets -> reports -> settings.
- Long task count if available through browser PerformanceObserver.
- JS heap size if available in WebView2/Chromium.
- React smoke success/failure and console errors.
- Normal and reduced-motion screenshots after theme changes.

Host / bridge:

- `state.snapshot`, `config.get`, `reports.list`, `targets.get` response time.
- Async action acceptance time for `monitor.start`, `monitor.stop`, `processes.refresh`, `reports.regenerate`, `diagnostics.generate`.
- WebView2 process failure count.
- Bridge request timeout count.

Watcher:

- Existing `watcher-poll-ms` with active monitor count and completed run count.
- Poll interval drift.
- Count of process enumerations per minute.
- Watcher CPU and working set during idle scanning.

Monitor-session:

- Time from target detected to PresentMon/samplers started.
- PresentMon start result and stderr.
- Sampler startup time.
- Stop time after target exit.
- Residual process count after stop.

Samplers:

- `FrameScopeProcessSampler.exe` CPU/working set/IO during a synthetic run.
- `FrameScopeSystemSampler.exe` CPU/working set/IO before and after per-core telemetry.
- `cpu-core-samples.csv` row count per minute.
- Sampling loop duration versus configured interval.

Report rendering:

- Report generation time.
- `framescope-interactive-data.js` size.
- `framescope-interactive-manifest.json` row counts.
- HTML open/render time for a representative large run.
- Chart switch/render times for existing chart tabs.

### Baseline Commands / Gates

Implementation phase should run these before and after performance changes:

- `tools\Run-Frontend.ps1 verify`
- `FrameScopeMonitor.exe --web-ui-smoke`
- `FrameScopeMonitor.exe --web-ui-smoke --web-ui-reduced-motion`
- C# test rebuild and targeted tests:
  - `tests\FrameScopeConfigStoreTests.exe`
  - `tests\FrameScopeWebBridgeTests.exe`
  - WebView2 runtime tests
  - sampler/report tests affected by telemetry
- Synthetic monitor session with a short-lived target or simulator.
- Residual process check for watcher, monitor-session, samplers, report generator, and bundled PresentMon.

No performance optimization should land unless:

- baseline exists
- after metrics exist
- existing behavior tests pass
- any regression is explained and accepted

### Candidate Optimizations After Baseline

Only evaluate after baseline:

- React: memoize expensive page rows/lists where measured re-render churn exists.
- React: avoid full config serialization on every render if it appears in profiles.
- React: keep theme resolution in one hook to avoid repeated DOM writes.
- WebView2 host: avoid noisy logging in normal mode; keep detailed logs behind existing diagnostics flag.
- Bridge: avoid repeated disk reads for snapshot if baseline shows it is a bottleneck, but preserve correctness after config save.
- Watcher: reduce stale report recovery frequency if idle baseline shows expensive scans.
- Samplers: keep per-core telemetry at slow interval; never run per-core voltage probes at high frequency.
- Report: preserve complete data, optimize only parsing/render/cache paths.

## 验收标准 / Acceptance Criteria

### Theme

- `ThemeMode=light` renders light tokens after app restart.
- `ThemeMode=dark` renders dark tokens after app restart.
- `ThemeMode=system` follows Windows/browser color scheme.
- Settings save failure keeps the user's draft.
- Existing mock preview still labels mock/live bridge correctly.
- Reduced motion still collapses motion durations and has no mixed/blank route frames.

### Settings / Config

- Old config without new fields loads successfully.
- Saving old config writes normalized new fields.
- Invalid theme/close/telemetry values normalize safely.
- `targets.save` preserves new global fields.
- `config.save` still rejects frontend-provided path/configPath.
- `framescope-config.example.json` documents the new fields.

### Tray

- Window X with minimize-to-tray hides the window and leaves app running.
- Tray Show window restores and focuses the WebView2 window.
- Tray Exit closes app.
- Active monitoring is not silently stopped by window X.
- Confirmed explicit exit stops FrameScope-owned watcher/monitor-session/samplers/PresentMon processes.
- Cancelled exit leaves monitoring and tray/window state unchanged.
- No duplicate tray icons after repeated hide/show.
- Smoke mode and automated close paths are not trapped in tray state.

### CPU Telemetry

- Existing `system-samples.csv` remains compatible.
- `cpu-core-samples.csv` is created when enabled and target is running.
- Disabled per-core telemetry does not create noisy artifacts.
- Missing per-core counters or voltage provider does not fail monitor-session.
- Voltage unavailable is explicit, not fabricated.
- Manifest records telemetry counts/availability.
- No new CPU telemetry appears in chart UI in this pass.

### Performance

- Baseline artifacts exist before optimization commits.
- After artifacts use the same metrics as baseline.
- WebView2 smoke and reduced-motion smoke pass.
- Watcher idle overhead does not regress beyond accepted threshold.
- System sampler overhead with per-core telemetry is measured and acceptable.
- Report generation/rendering still preserves full diagnostic data.
- Residual process check passes after monitor stop and after app exit.

### Regression Tests

Implementation should run, at minimum:

- frontend verify: `tools\Run-Frontend.ps1 verify`
- frontend tests covering design/interaction/motion contracts
- C# test rebuild: `tests\Build-FrameScopeTests.ps1`
- `FrameScopeConfigStoreTests.exe`
- `FrameScopeWebBridgeTests.exe`
- `FrameScopeWebView2RuntimeTests.exe`
- sampler/report tests affected by CPU telemetry
- `FrameScopeMonitor.exe --web-ui-smoke`
- `FrameScopeMonitor.exe --web-ui-smoke --web-ui-reduced-motion`
- synthetic monitor-session validation
- `git diff --check`
- residual process check

## Risk Notes

- Voltage is the highest uncertainty. It should be implemented as optional availability, not as a promised metric.
- Tray behavior can hide active monitoring. The tray icon and confirmation rules must be reliable before this ships.
- Default launch currently depends on WebView2 runtime. Do not blur runtime failure with tray or theme behavior.
- Performance work must not delete or downsample raw diagnostic data needed for troubleshooting.
- Existing workspace has unrelated modified files. Implementation should isolate commits and avoid reverting user or prior-agent changes.
